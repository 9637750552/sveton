## Ручные замечания пользователя

Пользователь бегло просмотрел часть review-файла и подтвердил проблемы качества:

- `1. doc_003_chunk_0001_stmt_001` - "Краткое интервью:" является заголовком, а не утверждением;
- `13. doc_003_chunk_0001_stmt_013` - "Комментарий:" является заголовком/служебной меткой, а не утверждением;
- `19. doc_003_chunk_0001_stmt_019`, `22. doc_003_chunk_0001_stmt_022`, `23. doc_003_chunk_0001_stmt_023` - дублируют смысл про инструмент и владение инструментом;
- `30. doc_003_chunk_0001_stmt_030`, `31. doc_003_chunk_0001_stmt_031` - "Удовлетворительные" / "Неудовлетворительные" непонятны без табличного контекста.

Эти замечания подтверждают, что v3 должен отдельно обрабатывать заголовки, дубли и таблицы.

### 1. Заголовки превращены в утверждения

Примеры:

- `doc_003_chunk_0001_stmt_001`: "Краткое интервью:";
- `doc_003_chunk_0001_stmt_005`: "Задание для переборки щита:";
- `doc_003_chunk_0001_stmt_013`: "Комментарий:";
- `doc_005_chunk_0001_stmt_004`: "Настройки:";
- `doc_005_chunk_0001_stmt_008`: "Кабель-трасса.";
- `doc_005_chunk_0001_stmt_011`: "Особенности.";
- `doc_006_chunk_0005_stmt_001`: "Фото элементов системы:".

Оценка: такие элементы должны быть контекстом/разделами, а не атомарными утверждениями.

### 2. Формулировки слишком шаблонные и слабые

Много утверждений построено по шаблонам:

- "В офисном обучении монтажника нужно отработать или оценить: ...";
- "В отчете по монтажу нужно указать или учитывать: ...";
- "Для фотофиксации монтажа нужно выполнить требование: ...".

Оценка: часть таких утверждений формально валидна, но качество базы знаний низкое. Нужна нормальная нормализация смысла, а не механическая подстановка цитаты в шаблон.

### 3. Не соблюдена атомарность в длинных пунктах

Примеры:

- `doc_004_chunk_0001_stmt_006` объединяет информирование начальника МО, фотофиксацию, описание по итогу выезда, озвучивание суммы заказчику и подтверждение ознакомления.
- `doc_004_chunk_0001_stmt_008` объединяет дополнительный осмотр, замеры, фото, возможные работы сейчас, перенос остальных работ и согласование через начальника МО.

Оценка: эти пункты нужно дробить на несколько атомарных утверждений.

### 4. Coverage gate не поймал пропуски

В `doc_005_chunk_0001` есть пункт:

- `Напряжение внешней сети -`

Но отдельного утверждения по нему в batch 002 нет, при этом `coverage_warnings = 0`.

Оценка: текущий coverage detector недостаточно надежен. Он пропускает строки без bullet/нумерации и считает покрытие полным.

### 5. Таблицы обработаны шумно

В `doc_003_chunk_0001` HTML-таблица контрольного листа дала отдельные утверждения по ячейкам:

- `Владение инструментом`;
- `Знание инструмента`;
- `Бережливость`;
- `Уверенность`;
- `Удовлетворительные`;
- `Неудовлетворительные`.

Оценка: таблицы нужно обрабатывать структурно. Характеристики должны быть привязаны к критерию, а не превращаться в отдельные независимые утверждения.

## Что при этом хорошо

- Цитаты являются точными подстроками источника.
- JSON-структура собирается корректно.
- Рекрутинговый файл действительно исключен.
- Coverage summary технически сохраняется.
- Короткие конкретные фото-требования в целом извлекаются полезно.

## Решение

Batch 002 не продвигать в canonical `atomic_statements.jsonl`.

Этот batch считать диагностическим прогоном, который выявил следующие требования к следующей версии extraction:

- не извлекать заголовки и section labels как самостоятельные утверждения;
- добавить `source_item_type`: `heading`, `instruction`, `requirement`, `table_row`, `example`, `note`, `empty`;
- разрешить `skipped_source_items` для заголовков с причиной `heading`;
- усилить правило атомарности длинных пунктов: один пункт может давать несколько statements;
- добавить table-aware extraction для HTML-таблиц;
- добавить coverage detector для коротких строк без bullet/нумерации, если они находятся внутри шаблона;
- добавить quality warning для шаблонных формулировок вроде "нужно указать или учитывать: ...";
- перезапустить batch 002 после доработки prompt/schema/runner.

## Следующий шаг

Перед следующим batch нужно доработать extraction pipeline v3:

1. Prompt: запретить утверждения из одних заголовков.
2. Schema: добавить source item typing или хотя бы расширить skipped reasons.
3. Runner: усилить quality warnings по заголовкам, HTML/table fragments и шаблонным формулировкам.
4. Повторить batch 002 и сравнить качество.

## Статус v3

Доработка v3 внесена после ручных замечаний пользователя:

- prompt запрещает извлекать `Краткое интервью:`, `Комментарий:` и похожие заголовки как утверждения;
- `skipped_source_items` теперь требует `source_item_type`;
- добавлены причины пропуска `heading`, `table_header`, `context_only`;
- runner добавляет quality warnings для заголовков, шаблонных формулировок, HTML/table fragments, табличных значений без контекста, явных дублей и длинных вероятно неатомарных утверждений;
- detector source items дополнительно учитывает короткие строки-поля без bullet, например `Напряжение внешней сети -`.

Следующий batch 002 нужно пересоздать заново; старые prompt-пакеты и raw-ответы diagnostic batch 002 не являются v3-прогоном.
