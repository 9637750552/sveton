## 1. Цель

Создать переиспользуемый слой семантического анализа, который превращает документы, интервью, регламенты и визуальные материалы в базу атомарных утверждений со ссылками на источники.

Первый прикладной корпус: документы для электриков и монтажников "Светон".

Долгосрочная цель: вынести слой в отдельный продукт или модуль, который можно применять в следующих проектах для анализа текстовых материалов.

Связанный будущий корпус: интервью с руководством компании. Для него используется тот же инженерный подход semantic extraction, но отдельная схема утверждений и отдельный эпик: [Эпик: Business / Sales / Commercial KB на основе интервью с руководством](../leadership_interviews_semantic_analysis/EPIC_LEADERSHIP_INTERVIEWS_SEMANTIC_ANALYSIS.md). Этот корпус нужен в первую очередь для отдельной Business / Sales / Commercial KB: описания бизнес-модели, продаж, коммерции, обучения продавцов/менеджеров/специалистов, стратегии, сайта, презентаций и маркетинговых материалов; связь с базой электриков является вторичной.

Обязательный референс-проект: `C:\Projects\semantic_extraction`. В этом проекте уже выполнялся семантический анализ интервью по продукту "Смородин"; его пайплайн, промпты, схемы, ошибки и удачные решения нужно изучить до финальной архитектуры нового слоя.

## 2. Проблема

Нельзя просить модель одним проходом "сделать базу знаний" из большого корпуса документов. При таком подходе модель:

- теряет локальный контекст;
- склеивает разные утверждения;
- обобщает там, где нужны точные правила;
- может добавлять выводы, которых нет в источнике;
- плохо сохраняет связь с конкретным документом.

Поэтому нужен пайплайн: извлечение текста, разбиение, атомарные утверждения, ссылки на источники, дедупликация, кластеризация и ручная проверка.

## 3. Принципы

- Каждое утверждение должно иметь источник.
- Утверждение должно быть атомарным: одна мысль, одно правило, одно требование.
- Модель не должна добавлять знания вне исходного текста.
- Инференсы и догадки не смешиваются с подтвержденными утверждениями.
- Дубли не удаляются без сохранения связи с несколькими источниками.
- Противоречия не сглаживаются, а выносятся на проверку.
- Итоговые инструкции пишутся только после семантического слоя.

## 4. Целевая архитектура

```text
Source files
  -> Text extraction
  -> Image extraction and inventory
  -> Structural parsing
  -> Document-aware chunking with overlap
  -> Atomic statement extraction
  -> Validation gate
  -> Statement validation
  -> Deduplication
  -> Clustering
  -> Contradiction detection
  -> Human review
  -> Knowledge base export
```

Корректировка после анализа `C:\Projects\semantic_extraction`: для документов электриков основным способом chunking должна быть структура документа, а не LLM-разметка смысловых переходов. LLM не управляет пайплайном; код задает границы, форматы, валидацию, метрики и экспорт. Кластеризация выполняется только после базовой валидации утверждений.

## 6. Декларативная конфигурация анализа

Для повторного использования нужен небольшой "язычок" конфигурации, вероятнее всего YAML.

Пример:

```yaml
project: electricians_knowledge_base
language: ru

source_types:
  - docx
  - pdf

visual_sources:
  extract_images: true
  classify_images: true
  require_caption_before_statement_link: true
  image_types:
    - photo_example
    - diagram
    - table_image
    - ui_screenshot
    - checklist_visual
    - decorative_or_unclear

chunking:
  strategy: structure_with_overlap
  max_tokens: 1200
  overlap_tokens: 200
  preserve_headings: true

extraction:
  forbid_external_inference: true
  require_source_quote: true
  split_atomic: true
  output_format: json

statement_types:
  - rule
  - requirement
  - instruction_step
  - checklist_item
  - recommendation
  - prohibition
  - definition
  - qualification_criterion
  - interview_signal

roles:
  - installer
  - electrician
  - manager
  - hq_engineer
  - project_lead

review:
  default_status: extracted
  require_human_review_for:
    - contradiction
    - low_confidence
    - safety_related
    - visual_only_source
```

## 7. Схема атомарного утверждения

Минимальная структура:

```json
{
  "id": "stmt_000001",
  "statement": "Монтажник должен сделать фотофиксацию критических точек монтажа.",
  "statement_type": "requirement",
  "role": "installer",
  "topic": "photo_report",
  "source_file": "example.docx",
  "source_document_id": "doc_001",
  "source_chunk_id": "chunk_001",
  "related_image_ids": ["img_001"],
  "source_quote": "Заполненный Чек-лист №2 с фотофиксацией 8 критических точек.",
  "confidence": "high",
  "review_status": "extracted",
  "notes": ""
}
```

Расширенные поля:

- `scope`: когда утверждение применяется;
- `condition`: при каком условии правило действует;
- `action`: что нужно сделать;
- `object`: к чему относится действие;
- `risk_level`: обычное, важное, safety-critical;
- `normalized_terms`: нормализованные термины;
- `related_image_ids`: изображения, которые иллюстрируют или уточняют утверждение;
- `visual_source_ids`: изображения, если утверждение требует визуальной проверки;
- `related_statement_ids`: связи с похожими утверждениями;
- `contradicts_statement_ids`: возможные противоречия.
