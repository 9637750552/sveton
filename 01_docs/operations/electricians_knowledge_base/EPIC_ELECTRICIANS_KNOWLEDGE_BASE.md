## 1. Цель

Создать проверяемую базу знаний для электриков, монтажников и менеджеров "Светон", на основе которой можно готовить:

- инструкции для монтажников;
- чек-листы для объекта и сервисного выезда;
- учебные материалы и аттестацию;
- скрипты интервью и отбора электриков;
- материалы сопровождения партнеров;
- основу для будущего ассистента/API.

База знаний не должна быть пересказом документов "в лоб". Каждое важное знание должно иметь связь с источником: документом, фрагментом текста, страницей, разделом или изображением.

## 2. Ключевое решение

Используем двухслойную архитектуру:

1. **Редакционный источник истины**: Markdown/документы в репозитории.
2. **Исполняемая база для ассистента**: PostgreSQL на сервере, позже, после стабилизации структуры.

На первом этапе база знаний живет в файлах проекта. PostgreSQL не заменяет редакционные документы, а индексирует и обслуживает утвержденные знания для поиска, API, бота и интерфейсов.

## 3. Связанный эпик

Этот эпик зависит от семантического слоя извлечения знаний:

[Эпик: Семантический слой извлечения знаний](../semantic_analysis_engine/EPIC_SEMANTIC_ANALYSIS_ENGINE.md)

Семантический слой нужен, чтобы из исходных DOCX/PDF извлекать атомарные утверждения со ссылками на источники, делать дедупликацию, кластеризацию и выявлять противоречия.

При выполнении каждой задачи обязательно применять проектное правило выбора уровня модели: [GUIDE.md, раздел 5](../../governance/GUIDE.md). В каждом следующем шаге нужно явно указывать: средняя, высокая, самая высокая или специализированная модель, и коротко объяснять выбор.

### Фича 3. Визуальный слой исходников

**Задача 3.1. Извлечь изображения из DOCX и PDF**

Описание:
Извлечь все полезные изображения из исходных документов и сохранить их отдельно от текста. Для DOCX сохранять встроенные картинки. Для PDF сохранять страницы как изображения, если отдельное извлечение объектов дает плохой результат или документ фактически является слайдовой инструкцией.

Целевая структура:

```text

  raw/
  normalized/
  inventory.md
```

Чек-лист:

- [ ] Создана папка `images/raw/`.
- [ ] Создана папка `images/normalized/`.
- [ ] Из DOCX извлечены встроенные изображения.
- [ ] Для PDF сохранены страницы или извлеченные изображения.
- [ ] Имена файлов включают идентификатор исходного документа.
- [ ] Сохранена связь изображения с исходным документом.

Критерии приемки:

- Изображения не остаются скрытыми внутри DOCX/PDF.
- Каждую картинку можно связать с исходным документом.

**Задача 3.2. Создать инвентаризацию изображений**

Описание:
Создать `images/inventory.md` и описать каждую картинку как отдельный источник знаний или вспомогательный материал.

Поля инвентаризации:

- `image_id`;
- `source_file`;
- `source_document_id`;
- `source_page` или `section_path`;
- `original_path`;
- `normalized_path`;
- `nearby_text`;
- `caption`;
- `image_type`;
- `topic`;
- `roles`;

- `table_image`: таблица как картинка;
- `ui_screenshot`: скриншот интерфейса или настроек;
- `checklist_visual`: визуальная памятка;
- `decorative_or_unclear`: непонятная, декоративная или нерелевантная картинка.

Чек-лист:

- [ ] Все извлеченные изображения внесены в `images/inventory.md`.
- [ ] У каждого изображения есть `image_id`.

## 6. Риски

- Модель может фантазировать при анализе больших документов без чанков и источников.
- Старые документы могут противоречить друг другу.
- Части PDF могут плохо извлекаться.
- Изображения могут содержать важную техническую информацию, которая теряется при текстовом извлечении.
- Картинки без подписи могут провоцировать неверные выводы модели.
- Некоторые инструкции могут быть устаревшими относительно реальной практики ГО.
- Нельзя сразу превращать все в Postgres без редакционной проверки.
