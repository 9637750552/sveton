# Руководство по стандартам и процессам (GUIDE)

## 1. Философия модели "Инженерный Поставщик"
Мы продаем не товар, а **услугу гарантированного энергоснабжения**. Наша ценность не в "железе", а в:
1. Экспертном расчете нагрузок.
2. Гарантии совместимости (Штиль + Бинеос).
3. Удаленном контроле качества монтажа.

## 2. Правила удаленного взаимодействия (Acceptance)
Любой монтаж, выполненный сторонним партнером, считается принятым ТОЛЬКО при выполнении 3-х условий:
1. Заполненный Чек-лист №2 с фотофиксацией 8 критических точек.
2. Видео-подтверждение работы системы в режиме "Авария" (имитация отключения сети).
3. Дистанционная проверка настроек инвертора через видеосвязь инженером "Светон-Юг".

## 3. Принципы "Красных линий"
Мы сразу отказываем клиенту в монтаже (или требуем платной подготовки), если:
*   В щитке невозможно выделить резервную группу без демонтажа отделки.
*   Отсутствует заземление (и клиент не готов его делать).
*   Пусковые токи оборудования превышают номинал ИБП более чем в 3 раза.

## 4. Протокол "Swap-сервис"
При поломке оборудования в гарантийный период:
1. Удаленная диагностика.
2. Отгрузка подменного блока курьерской службой.
3. Замена блока силами клиента или местного партнера.
4. Возврат брака в ГО (Москва) для экспертизы.

## 5. Правило выбора уровня модели

Для каждой новой задачи, следующего шага, эпика или крупного решения исполнитель должен явно указывать рекомендуемый уровень модели:

- **Средняя модель**: рутинная обработка, извлечение текста, техническая подготовка файлов, первичная инвентаризация, массовые однотипные операции по строгой схеме.
- **Высокая модель**: проектирование структуры, редакционная сборка, анализ противоречий, написание инструкций, чек-листов, интервью, оценка качества результата.
- **Самая высокая модель**: архитектура семантического слоя, схема базы знаний/PostgreSQL, критичные промпты извлечения, методология контроля фантазий, финальное ревью ключевых решений.
- **Специализированная/предыдущая модель или версия**: использовать, если для конкретной задачи она доказала лучшую практическую эффективность и не уводит работу в лишние теоретические рассуждения.

Формат рекомендации:

```text
Рекомендуемый уровень модели: <средняя / высокая / самая высокая / специализированная>.
Почему: <короткое обоснование>.
```

Порядок исполнения:

1. Сначала исполнитель формулирует следующий шаг и рекомендуемый уровень модели.
2. Затем исполнитель ждет явного подтверждения руководителя, что выбранная модель/уровень подходят и можно приступать.
3. Только после подтверждения исполнитель начинает выполнять задачу.

Запрещено начинать новый существенный шаг сразу после собственной рекомендации модели. Исключение: мелкие правки самого проектного правила, если руководитель прямо попросил изменить правило.

Если задача состоит из нескольких этапов, уровень модели указывается по этапам. Например: средняя модель для массового извлечения утверждений, высокая модель для проверки выборки, самая высокая модель для финальной архитектуры и критериев качества.

Правило по умолчанию: не использовать самую высокую модель для рутинной массовой обработки, если качество обеспечивается строгой схемой, chunking, source quotes и последующим ревью. Самую высокую модель применять там, где цена ошибки высока: архитектура, методология, финальные решения, безопасность и контроль качества.

## 6. Методология сборки разделов базы знаний

Source-backed разделы базы знаний собираются единым способом. Нельзя менять формат сборки от раздела к разделу без отдельного изменения методологии.

Обязательные правила:

1. Факты в редакционном разделе берутся только из `atomic_statements.jsonl` и соответствующего canonical-кластера.
2. Изображения для редакционного раздела берутся только из утвержденных связей `statement_images.jsonl`.
3. Изображения не создают новые факты и не заменяют semantic extraction. Они только иллюстрируют уже извлеченные canonical-утверждения.
4. Визуальные примеры вставляются рядом с соответствующим смысловым подразделом и связанными утверждениями.
5. Формат визуального примера должен повторять существующие разделы: изображение, `image_id`, затем связи вида `statement_id -> image_id`.
6. Нельзя добавлять отдельный верхнеуровневый каталог или блок "визуальный слой" внутри одного раздела, если остальные разделы используют inline visual examples.
7. Ручные image-links допустимы только как отдельный явно задокументированный pass. Их нельзя незаметно смешивать с extraction-backed связями.
8. Если для будущего раздела нужен иной способ визуальной подачи, сначала меняется эта методология, затем пересобираются разделы по новому правилу.

## 7. Beans как task tracker проекта

В этом проекте локальный task tracker ведется через `Beans`.

Перед началом инженерной работы исполнитель должен запускать:

```bash
beans prime
```

И учитывать выданные им инструкции как часть рабочего контура проекта.

Правила работы:

1. Для новой инженерной задачи создавать `bean`, если уже нет подходящего существующего.
2. Для коротких консультаций, быстрых ответов и анализа без изменения файлов `bean` можно не создавать.
3. Для маленького локального исправления можно использовать существующий `bean` или создать короткий `bug`-`bean`, если исправление должно сохраниться в истории проекта.
4. В процессе работы нужно держать checklist в `bean` актуальным: выполненные пункты переводить из `- [ ]` в `- [x]`.
5. Переводить `bean` в статус `completed` можно только когда checklist реально закрыт.
6. Если задача велась через `Beans`, изменения в `.beans/` коммитятся вместе с кодом, данными и документацией этой задачи.

Рекомендуемые типы:

- `milestone` - крупная контрольная точка.
- `epic` - группа связанных работ.
- `feature` - пользовательская или процессная возможность.
- `task` - конкретная техническая задача.
- `bug` - исправление дефекта.

Проектный prefix для `Beans`: `Sv-`.

## 8. Границы трех проектов

Проект разделяется на три логических project boundaries:

1. `Sveton` - бизнес-проект, входящие документы, стратегия, governance и доменный контекст.
2. `Semantic Analysis Engine` - переиспользуемый продукт для semantic extraction, chunking, validation, clustering и review workflow.
3. `Sveton Knowledge Base` - собранная база знаний с canonical statements, clusters, source links, image links, safety review и editorial sections.
