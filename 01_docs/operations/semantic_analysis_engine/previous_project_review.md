## 3. Как был устроен старый пайплайн

Старый проект строился как последовательный пайплайн шагов:

1. Подготовка стенограммы в канонический CSV.
2. Формирование перекрывающихся окон.
3. LLM-разметка `SHIFT / CONTINUE / UNCLEAR`.
4. Агрегация SHIFT-меток.
5. Валидация границ.
6. Формирование смысловых чанков.
7. LLM-извлечение claims по чанкам.
8. Метрики качества и рекомендации по параметрам.
9. Дополнительное переписывание соседних чанков в claims с низкой уверенностью.
10. Слияние claims в общий корпус.
11. Кластеризация через embeddings + HDBSCAN.
12. LLM-подписи кластеров.
13. Сборка skeleton/raw-документов на основе кластеров.

Ключевой архитектурный принцип: LLM не управляет пайплайном. Код задает границы, форматы, валидирует артефакты и считает метрики. LLM используется только в ограниченных местах: перевод, SHIFT-разметка, извлечение claims, переписывание/подписи.

## 5. Что нельзя переносить без изменений

Старый проект был заточен под интервью. Для документов, инструкций и регламентов электриков нужны другие акценты.

Не переносим напрямую:

- SHIFT-разметку как основной способ chunking. В документах сначала нужно использовать структуру: заголовки, списки, таблицы, страницы, номера пунктов.
- CSV-only модель данных. Для документов лучше использовать JSONL для chunks/statements, потому что поля станут вложенными: source spans, quotes, roles, risk flags, related statements.
- Claims без source quote. Для базы знаний каждое утверждение должно иметь цитатную опору.
- Лимит "до 5 claims на чанк" как жесткое правило. Для чек-листа или таблицы в одном чанке может быть больше атомарных пунктов.
- "Dumb merge" без upstream-валидации. Для базы знаний нельзя пропускать мусорные утверждения дальше по пайплайну.
- Переписывание соседних чанков как отдельный источник claims без строгой пометки низкой уверенности. Для регламентов это может размыть норму.
- Кластеризацию как замену дедупликации. Кластеризация помогает найти группы, но canonical statement и связи с источниками нужно формировать отдельно.

## 6. Найденные ограничения старого подхода

- Claim мог быть неполным: в `claims_all.csv` встречается пример `В России около 6.`, который прошел до общего корпуса и был помечен как noise только на этапе кластеризации.
- Часть чанков возвращала `NONE` из-за шума, обрывков и слабой структуры интервью. Для документов это менее критично, но PDF с плохим layout может создать похожую проблему.
- `claims_cleaning.py` делал только нормализацию whitespace и точную дедупликацию внутри чанка. Этого недостаточно для базы знаний.
- В `claims` не было обязательной цитатной опоры, типа утверждения, роли, уровня риска и review status.
- Старый Step 9 специально не валидировал содержание. Для базы знаний нужен отдельный validation gate до merge/cluster.
- Кластеризация хорошо группирует похожие claims, но не решает противоречия и условия применимости.

## 7. Корректировка подхода для документов электриков

Новый semantic layer должен строиться так:

```text
Raw DOCX/PDF
  -> extracted text
  -> document-aware chunks
  -> atomic statements with source_quote
  -> validation gate
  -> canonical statements / duplicates
  -> clusters
  -> contradictions
  -> human review
  -> knowledge base export
```

Отличия от старого проекта:

- Основной chunking: структура документа, а не SHIFT-разметка.
- Overlap сохраняем, но как технический контекст вокруг заголовков/списков.
- У каждого чанка должны быть `source_document_id`, `source_file`, `section_path`, `page_start/page_end`, `text`, `previous_chunk_id`, `next_chunk_id`.
- У каждого statement должны быть `statement`, `source_quote`, `type`, `role`, `topic`, `risk_level`, `confidence`, `review_status`.
- Safety-related темы, например УЗО и щиты, автоматически получают `review_required`.
- PDF с шумным layout не запускаются в production extraction без дополнительной проверки или отдельного extractor-а.
- Merge-этап должен сохранять все источники, но не должен скрывать противоречия.
- Дедупликация должна создавать canonical statement и список supporting sources.
- Кластеризация должна работать после базовой валидации утверждений, а не вместо нее.

## 8. Решение по моделям

Рекомендуемый режим:

- `5.3 / средний`: массовая техническая подготовка, извлечение текста, первичный chunking, пересчет метрик.
- `5.4 / высокий`: массовое извлечение атомарных утверждений по жесткой JSON-схеме, если схема уже проверена.
- `5.5 / самый высокий`: проектирование схемы, prompt extraction, validation rules, финальное ревью подхода, разбор противоречий и safety-related тем.

Для первого запуска на корпусе электриков:

- Спроектировать схему и prompt на `5.5`.
- Протестировать 3-5 чанков на `5.5`.
- Массовый прогон делать на `5.4`, если качество теста приемлемое.
- Выборочную проверку и корректировки делать на `5.5`.
